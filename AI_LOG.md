## Commit 1 — CI pipeline + hermetic harness + Allure wiring

### Discovery: `allure-go` results path semantics differ from the plan's CI sketch
- **Context:** The plan's CI sketch set `ALLURE_RESULTS_PATH: allure-results` and `mkdir -p allure-results`,
  implying results land directly in `./allure-results`.
- **What the agent found (by reading the installed lib source, not the plan):** in
  `dailymotion/allure-go@v0.9.0`, `init.go`/`attachment.go` compute
  `resultsPath = fmt.Sprintf("%s/allure-results", os.Getenv("ALLURE_RESULTS_PATH"))`.
  The library *appends* `allure-results` itself. So `ALLURE_RESULTS_PATH=allure-results`
  would write to `allure-results/allure-results`.
- **Compounding issue:** `go test ./...` runs each package with its working directory set to that
  package's dir, so results would land in `test/allure-results`, not the repo root — breaking the
  CI report step that looks at `allure-results`.
- **Fix:** set `ALLURE_RESULTS_PATH` to the **repo root absolute path** (`${{ github.workspace }}` in CI,
  `$(pwd)` locally) so results always land in `<root>/allure-results` regardless of the test binary's cwd.

### Discovery + course-correct: fresh checkout doesn't compile, and the agent's first fix was overkill
- **Found:** `main.go` imports `github.com/alpody/fiber-realworld/docs`, a package generated by
  `swag init`. `/docs` is gitignored, so a fresh clone cannot `go build ./...` or `go test ./...`
  (the `main` package fails to load).
- **Agent's first instinct:** generate the docs in CI (install the `swag` CLI, run `swag init`). This
  worked but was the wrong call — it drags a swagger toolchain into the test pipeline just to make an
  unrelated `main` package compile.
- **Human override #1:** "why do you need swagger now?" — correct. The test packages never import `docs`.
- **Agent's second attempt:** scope the test run to exclude the `main` package via
  `go test $(go list -e ./... | grep -v '/fiber-realworld$')`. Worked, but the human pushed again —
  "all the tests in /handler work locally, why are you excluding anything?" — fair: a package-filter in
  the CI command is a smell that hides a real build problem.
- **Human override #2 + their fix:** the human commented out the unused
  `_ "github.com/alpody/fiber-realworld/docs"` import in `main.go` (the only thing pulling in the
  generated package). `main` now compiles on a clean checkout, so CI is just plain `go test ./...` —
  no filter, no swagger, no go.mod pollution. The simplest thing that builds.

### Human override: drop the `integration` build tag entirely
- **Agent's design (from the plan):** put the suite behind `//go:build integration` so the "fast unit
  layer" could run standalone and the "slow DB suite" was opt-in.
- **Human's call:** "I don't think we need that integration tag — all tests must run on CI, otherwise
  what's the point?" Correct: the suite is backed by **in-memory** SQLite and runs in ~0.1s/test, so
  the slow/opt-in justification doesn't hold. A suite that doesn't run in CI protects nothing, and the
  tag only added friction.
- **Fix:** removed the build tag from all `test/` files; the suite now runs under plain `go test`.

### Trap the tag created along the way (now moot, kept as a lesson)
- While the tag still existed, the agent's first CI command put `-tags=integration` on `go test` but
  **not on `go list`**. Without the tag, `go list ./...` omits a fully build-tagged package, so the
  filtered package list never contained `.../test` — `go test` would have run only the legacy handler
  tests and reported green: the classic "tests pass because they never ran". Removing the tag removes
  the whole footgun (the `test` package is now always listed). General lesson: build tags + package
  filtering interact in non-obvious ways; the simplest design that still runs everything wins.

### Decision (from the env semantics above): pin ALLURE_RESULTS_PATH to an absolute root
- Because `go test ./...` runs each package binary with cwd = that package's directory, a relative
  `ALLURE_RESULTS_PATH` would scatter results under `test/allure-results`. CI sets it to
  `${{ github.workspace }}` (absolute) so every binary writes to one `<root>/allure-results`.

### Found: allure-go's empty-env fallback floods a plain local run with errors
- **Symptom (human ran `go test` with no env set):** `environment variable ALLURE_RESULTS_PATH cannot
  be empty`, then `mkdir /allure-results: read-only file system`, then failed result writes.
- **Root cause:** when `ALLURE_RESULTS_PATH` is unset, allure-go doesn't skip reporting — it builds the
  path as `"" + "/allure-results"` = the absolute `/allure-results` and tries to create it at the
  filesystem root. The test still *passes* (the write failures are non-fatal logs), but the report is
  lost and the output is alarming.
- **Fix:** added `TestMain` to the test package that defaults `ALLURE_RESULTS_PATH` to the current
  working directory when unset, so a bare local `go test` writes `./allure-results` with zero setup.
  CI's explicit value (repo root) still wins. Also broadened `.gitignore` to match `allure-results/`
  anywhere, since a local run now writes `test/allure-results`.

### Correction: API constant name verified before use
- The plan flagged uncertainty between `allure.ApplicationJson` vs `allure.ApplicationJSON`.
- Verified against `mimetypes.go` in the installed version: it is `ApplicationJson` (lowercase `son`).
  Using the wrong casing would not compile. Captured so later commits don't guess.
